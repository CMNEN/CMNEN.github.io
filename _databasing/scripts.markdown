The EMu data entry template is specifically structured to support batch imports across multiple modules (e.g. Catalogue, Collection Events, Sites, Parties, and Taxonomy). Rather than entering records directly into EMu, data is first entered on an Excel data entry sheet (see above and "Data entry" on the side ribbion), compared with existing records on EMu, and finally transformed into import-ready sheets to be imported into EMu.

The data-standardization and EMu-checking steps are carried out with an R script. This helps speed up the workflow and also reduces errors associated with manual record matching. Simply put, data from the Excel entry sheet is first cleaned and standardized, then checked against existing EMu records to identify matches with existing records to retrieve IRNs (Internal Reference Numbers) of said records. Lastly, any remaining unmatched records are formatted into import-ready sheets, which can then be imported into EMu.

The following sections outline how the R script handles data, including how values are processed and which fields are used to match records across the different EMu modules.

## General order

You must run the Parties and Taxonomy sections of the script first, as these populate the IRN columns that will be linked to Collection Event and Catalogue records. Similarly, the section that checks for existing Site records must be completed before running the section that handles Collection Event records, since the value in "ColSiteRef.irn" is used when determining whether a Collection Event already exists (i.e. the associated Site record must already be known before the Collection Event can be checked).

In general, the data transformation script must be run in the following order, based on the type of record being processed (see Table 1 for the field dependencies underlying this workflow):

1. Parties
2. Taxonomy
3. Sites
4. Collection Events
5. Catalogue

## Parties and Taxonomy

Parties (names of the collectors and determiners as entered in the data entry sheet) and taxonomical names must be matched to their respective (existing) records before any Collection Event records or Catalogue records can be imported.

For Parties, this is done by simply taking the values from the columns <span style="background-color: #d2ecb6;">"Determiner"</span>, <span style="background-color: #d2ecb6;">"Determiner2"</span>, <span style="background-color: #d2ecb6;">"Determiner3"</span>, <span style="background-color: #f5e5ff;">"Collector"</span>, <span style="background-color: #f5e5ff;">"Collector2"</span>, <span style="background-color: #f5e5ff;">"Collector3"</span>, and <span style="background-color: #f5e5ff;">"Collector4"</span>, comparing them to a report of all party names in EMu that has been properly formatted into the "**First intial. Second intial. Last name**" format. All values that are not matched to an existing name are moved into an import sheet to be uploaded into EMu. This method of pulling IRNs is not perfect (e.g. handling duplicate names or records of people with the same initials and surname), but works well for Entomology where we often deal with the same "group" of people over thousands of specimens.

For Taxonomy records, the process is slightly more complex. Scientific names in the columns <span style="background-color: #d2ecb6;">"Scientific_name"</span> and <span style="background-color: #d2ecb6;">"Prev_scientific_name"</span> are checked against an EMu report for all existing taxonomy records. Values that do not match an existing name are piped into an import sheet as with new Party records. For unmatched species names, the genus (i.e. the first term in the binomial name) is also checked against existing taxonomy records. If the genus does not already exist in EMu, it is added to the import sheet so that the appropriate RanParentRef.irn can be populated when its associated new species record is created.

## Sites

For Site records, matching is based on a combination of fields comprised of locality descriptors and coordinate data. All relevant fields are pulled out from the data entry sheet while records in the data entry sheet are classified based on the type of coordinate information present (decimal or DMS; point or range). All data entry records are then matched against existing EMu Site records by comparing corresponding field values (see Table 1). Where a match is found, the corresponding existing Site record IRN is assigned. Matches are only accepted when all corresponding Site fields match exactly. Records that do not match existing Sites are separated out and moved into an import sheet. The fields “Country”, “Province/State”, and “County/Region” are used to populated the locality hierarchy table in the EMu Site record.

## Collection Events

Collection Event records are handled in a very similar way to Site records (data is extracted, matched against existing EMu Collection Event records, and non-matches are moved into an import sheet), with the main difference being that collector IRNs are also included in the matching process, as that is an import part of a Collection Event record (e.g. Who are the collectors of this beetle?). The columns ColParticipantRef_tab(1).irn through ColParticipantRef_tab(4).irn (if applicable) are added to the dataframe during the Parties record matching step.

## Catalogue 

Once all Parties, Taxonomy, Sites, and Collection Events have been processed, the specimen records themselves (Catalogue records) can finally be imported into EMu. Unlike the previous modules, no matching against existing Catalogue records is carried out during this step, as each specimen being databased is presumed to represent a unique, previously undatabased Catalogue record.
